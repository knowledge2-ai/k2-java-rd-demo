ANSWER A:
**Recommendation**
Anchor the answer on the Kafka Connect 4.2 source API Javadocs plus the runtime classes that actually implement the poll/commit loop. For standard source tasks, the key behavior is split between `AbstractWorkerSourceTask` and `WorkerSourceTask`; for exactly-once semantics, `ExactlyOnceWorkerSourceTask` is the parallel anchor.

**Implementation anchors**
- `connect/api/src/main/java/org/apache/kafka/connect/source/SourceTask.java` for the versioned contract:
  - `poll()` blocks but must return control regularly.
  - `commit()` is invoked periodically on offset commits and does not necessarily align with the latest `poll()` offsets.
  - `commitRecord()` is the fine-grained callback on successful delivery or filtered/dropped records.
- `connect/api/src/main/java/org/apache/kafka/connect/source/TransactionContext.java` for exactly-once transaction boundary semantics tied to `poll()`.
- `connect/runtime/src/main/java/org/apache/kafka/connect/runtime/AbstractWorkerSourceTask.java` for the shared poll-convert-send loop:
  - `execute()` drives `poll()` and `sendRecords()`.
  - `poll()` wraps retriable errors by returning `null`.
  - `sendRecords()` handles record dispatch, acks, drops, and commit callbacks.
  - `commitTaskRecord()` and `commitSourceTask()` wrap `SourceTask.commitRecord()` and `SourceTask.commit()`.
- `connect/runtime/src/main/java/org/apache/kafka/connect/runtime/WorkerSourceTask.java` for standard source-task offset commit behavior:
  - `beginSendIteration()` refreshes committable offsets.
  - `recordSent()` and `recordDropped()` feed commit tracking.
  - `commitOffsets()` drains committable offsets, flushes them through `OffsetStorageWriter`, and still calls `SourceTask.commit()` even when nothing is pending.
- `connect/runtime/src/main/java/org/apache/kafka/connect/runtime/ExactlyOnceWorkerSourceTask.java` for the EOS variant:
  - commits through producer transactions instead of the non-EOS flush path.

ANSWER B:
**Recommendation**
Anchor the answer on the Kafka 4.2.0 Connect docs plus the runtime commit scheduler, not just `WorkerSourceTask` itself. The clearest behavior split is:
- source tasks poll records in the task execution loop, and
- offset commits are driven by `SourceTaskOffsetCommitter`, which only calls `WorkerSourceTask.commitOffsets()` when `shouldCommitOffsets()` is true and logs/handles failures around the flush path [https://kafka.apache.org/42/kafka-connect/connector-development-guide/#part-0010](https://kafka.apache.org/42/kafka-connect/connector-development-guide/#part-0010), [repo://apache/kafka@4.2/connect/runtime/src/main/java/org/apache/kafka/connect/runtime/SourceTaskOffsetCommitter.java](repo://apache/kafka@4.2/connect/runtime/src/main/java/org/apache/kafka/connect/runtime/SourceTaskOffsetCommitter.java).

**Implementation anchors**
- `WorkerSourceTask` is the implementation class to inspect for the source-task execution path, including the commit-flush path and timeout/cancel handling [repo://apache/kafka@4.2/connect/runtime/src/main/java/org/apache/kafka/connect/runtime/WorkerSourceTask.java](repo://apache/kafka@4.2/connect/runtime/src/main/java/org/apache/kafka/connect/runtime/WorkerSourceTask.java).
- `SourceTaskOffsetCommitter` is the neighboring runtime class that schedules and gates commits; its `commit(WorkerSourceTask)` helper skips when there are no offsets to commit, otherwise invokes `workerTask.commitOffsets()` and catches unexpected exceptions so the scheduled executor keeps running [repo://apache/kafka@4.2/connect/runtime/src/main/java/org/apache/kafka/connect/runtime/SourceTaskOffsetCommitter.java](repo://apache/kafka@4.2/connect/runtime/src/main/java/org/apache/kafka/connect/runtime/SourceTaskOffsetCommitter.java).
