- Treat `DefaultCheckpointPlanCalculator` as the behavioral source of truth: it builds the checkpoint plan asynchronously on the JobMaster main executor, rejects finished-task checkpointing when disabled, validates task state, and then splits into the all-running vs partially-finished branches.

**Implementation anchors**
- `CheckpointPlanCalculator` defines the contract: calculate the next checkpoint plan, including trigger/wait/commit responsibilities.
- `CheckpointPlanCalculatorContext` supplies the main executor and the “has finished tasks” signal.
- `ExecutionGraphCheckpointPlanCalculatorContext` adapts `ExecutionGraph` into that contract.
- `DefaultCheckpointPlanCalculator` implements the actual plan logic, including `ALL_TO_ALL` / `POINTWISE` upstream checks, trigger selection, and finished-task collection.
- `DefaultExecutionGraph#createCheckpointPlanCalculator(...)` is where the implementation is wired into the execution graph / coordinator path.

**Tests to inspect or add**
- `DefaultCheckpointPlanCalculatorTest` is the direct spec for graph-shape cases, multi-input and multi-level topologies, and “task not RUNNING” failures.
- `DefaultCheckpointPlanTest` covers finished-task fulfillment and UnionListState edge cases after a plan has been computed.
- `CheckpointCoordinatorTriggeringTest` covers the coordinator path when checkpointing is allowed after a task has already finished.
- `CheckpointCoordinatorMasterHooksTest` and `FailoverStrategyCheckpointCoordinatorTest` are useful integration neighbors because they construct the calculator through the coordinator path.
- If adding coverage, the best regression target is the partially-finished branch in `calculateAfterTasksFinished()`, especially mixed `ALL_TO_ALL` / `POINTWISE` inputs with `allowCheckpointsAfterTasksFinished` on and off.
