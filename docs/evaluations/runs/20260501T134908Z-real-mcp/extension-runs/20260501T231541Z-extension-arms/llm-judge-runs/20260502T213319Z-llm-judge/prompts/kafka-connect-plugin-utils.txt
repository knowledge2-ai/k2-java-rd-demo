ANSWER B:
**Recommendation**
- Anchor the answer on the 4.2 source docs in [`WorkerConfig.java`](repo://apache/kafka@4.2/connect/runtime/src/main/java/org/apache/kafka/connect/runtime/WorkerConfig.java#L129-L154) for the `plugin.path` contract, and on [`PluginUtils.java`](repo://apache/kafka@4.2/connect/runtime/src/main/java/org/apache/kafka/connect/runtime/isolation/PluginUtils.java#L203-L443) for the actual scan/alias logic.
- For the two behaviors you asked about:
  - `plugin.path` is a comma-separated list of top-level plugin roots; the implementation walks directories, follows symlinks, accepts jars/zips or class hierarchies, and preserves class-only roots when there are no archives.
  - alias helpers use both `simpleName()` and `prunedName()`; `computeAliases()` keeps only unambiguous aliases and drops collisions.
