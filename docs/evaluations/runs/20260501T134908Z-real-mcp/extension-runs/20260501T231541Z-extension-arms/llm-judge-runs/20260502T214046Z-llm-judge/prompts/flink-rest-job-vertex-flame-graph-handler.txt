ANSWER B:
**Recommendation**
Use the runtime REST API generator as the version-pinned documentation anchor, then point to the job-vertex flame-graph handler and its headers as the implementation source of truth. In Flink 2.2.0, the endpoint is the `GET /jobs/:jobid/vertices/:vertexid/flamegraph` REST call, with optional `type` and `subtaskindex` query params, and it returns `terminated`, `waiting`, `disabled`, or a computed `VertexFlameGraph` depending on vertex state and sampling availability.

**Implementation anchors**
- Version-pinned docs: `RuntimeRestAPIDocGenerator` emits docs per `RuntimeRestAPIVersion`, and skips `V0`, so the pinned runtime docs are effectively the `V1` REST API docs. See `repo://apache/flink@release-2.2.0/flink-docs/src/main/java/org/apache/flink/docs/rest/RuntimeRestAPIDocGenerator.java#L62-L83` and `repo://apache/flink@release-2.2.0/flink-runtime/src/main/java/org/apache/flink/runtime/rest/versioning/RuntimeRestAPIVersion.java#L30-L34`.
- Docs regeneration policy: `flink-docs/README.md` explicitly says REST docs must be regenerated when handlers or referenced headers/parameters change. See `repo://apache/flink@release-2.2.0/flink-docs/README.md#L24-L36`.
- Endpoint contract: `JobVertexFlameGraphHeaders` defines the URL, GET method, `OK` response, and description. See `repo://apache/flink@release-2.2.0/flink-runtime/src/main/java/org/apache/flink/runtime/rest/messages/JobVertexFlameGraphHeaders.java#L27-L78`.
- Behavior implementation: `JobVertexFlameGraphHandler` handles subtask selection, terminal vertices, flame-graph type selection, sampling lookup, and fallback states. See `repo://apache/flink@release-2.2.0/flink-runtime/src/main/java/org/apache/flink/runtime/rest/handler/job/JobVertexFlameGraphHandler.java#L52-L197`.
- Endpoint wiring: `WebMonitorEndpoint` registers the handler and gates it behind `RestOptions.ENABLE_FLAMEGRAPH`. See `repo://apache/flink@release-2.2.0/flink-runtime/src/main/java/org/apache/flink/runtime/webmonitor/WebMonitorEndpoint.java#L856-L874`.
