ANSWER B:
**Recommendation**
Anchor the explanation on the source Javadocs and the decider itself, then verify with the unit test matrix in `CheckpointRequestDeciderTest`.

The scheduling rules in `CheckpointRequestDecider` are:

- Queueing is priority-based, not pure FIFO.
- Savepoints outrank checkpoints.
- Forced requests outrank non-forced requests.
- Non-periodic requests outrank periodic requests.
- Older timestamps win within the same priority class, with `identityHashCode` as a final tie-breaker.
- A new request is dropped immediately with `TOO_MANY_CHECKPOINT_REQUESTS` when the queue is already full of non-periodic requests.
- If the queue is over capacity after insertion, the lowest-priority request is evicted and failed with `TOO_MANY_CHECKPOINT_REQUESTS`.
- Execution is blocked while another trigger is already in progress, while the queue is empty, or while cleanup work is above the configured threshold.
- When pending checkpoints have reached `maxConcurrentCheckpoints`, only a forced request may execute; otherwise the request stays queued.
- Periodic, non-forced requests are additionally held back by `minPauseBetweenCheckpoints`; if the pause has not elapsed, that request is failed with `MINIMUM_TIME_BETWEEN_CHECKPOINTS` and the coordinator is asked to reschedule.
