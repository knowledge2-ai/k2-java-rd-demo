ANSWER A:
**Recommendation**
- Treat `PendingCheckpoint` as a two-phase lifecycle object: it starts pending, becomes fully acknowledged as task/coordinator/master acks arrive, then ends either in success (`CompletedCheckpoint`) or failure/discard.
- There is no explicit enum state machine in `PendingCheckpoint`; the effective state is encoded by `notYetAcknowledged*`, `disposed`, `discarded`, and the two completion futures.
- Success path: full ack -> `CheckpointCoordinator.maybeCompleteCheckpoint()` -> `PendingCheckpoint.finalizeCheckpoint()` -> metadata write -> `dispose(false)` -> coordinator stores and completes the `CompletedCheckpoint`.
- Failure path: timeout/decline/subsumed/other error -> `abort()` -> futures complete exceptionally -> `dispose(true)` -> state is cleared and later acks return `DISCARDED`.
- Savepoints are special: `canBeSubsumed()` is false for savepoints, and aborting a savepoint with `CHECKPOINT_SUBSUMED` is illegal.

**Recommendation**
