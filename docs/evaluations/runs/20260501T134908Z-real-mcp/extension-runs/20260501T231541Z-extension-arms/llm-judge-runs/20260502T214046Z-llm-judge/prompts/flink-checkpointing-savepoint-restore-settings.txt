ANSWER B:
**Recommendation**
- For a 2.2.0 upgrade, treat `SavepointRestoreSettings` as the job-submission contract for “resume from this savepoint.”
- Use `SavepointRestoreSettings.forPath(savepointPath, allowNonRestoredState, recoveryClaimMode)`:
  - `allowNonRestoredState=false` for strict upgrades where every prior operator/state mapping must still exist.
  - `allowNonRestoredState=true` only when the upgraded job intentionally drops or renames stateful operators and you want unmapped state skipped.
  - `recoveryClaimMode` controls how the savepoint is reclaimed on restore; `NO_CLAIM` is the safest default if you do not want the restore to consume the savepoint as a claimed snapshot.
- `none()` means no savepoint restore at all.
- The restore path is serialized into config keys via `StateRecoveryOptions`, then applied by runtime restore code only if checkpoint restoration did not already succeed.
