ANSWER A:
**Recommendation**
Anchor the answer on the REST doc generators and the request lifecycle, not on `FileUploads` alone. In Flink 2.2.0, uploaded-file tracking for REST requests is documented and exercised through `RuntimeOpenApiSpecGenerator`/`OpenApiSpecGenerator`, with the runtime path flowing `FileUploadHandler -> FileUploads -> AbstractHandler`.

**Implementation anchors**
- `FileUploads` is the container/scanner/cleanup type: it validates the upload directory, walks it to return uploaded files, and deletes the directory on `close()`.
- `FileUploadHandler` creates the per-request temp dir, stores `new FileUploads(currentUploadDir)` on the Netty channel, and exposes it through `getMultipartFileUploads(ctx)`.
- `AbstractHandler` pulls that attribute once, rejects unexpected uploads when `acceptsFileUploads()` is false, passes the files into `HandlerRequest`, and closes them after request processing.
- `JobSubmitHeaders` is the REST contract that declares multipart uploads and the `request` JSON attribute.

**Tests to inspect or add**
- Inspect `FileUploadsTest` for scan/empty/cleanup behavior.
- Inspect `AbstractHandlerTest.testFileCleanup` for lifecycle cleanup after request completion.
- Inspect `FileUploadHandlerITCase` for multipart success, malformed upload cleanup, unknown attribute cleanup, and filename handling.
- Inspect `RestServerEndpointITCase` for end-to-end upload acceptance/rejection and the no-file multipart case.
- Inspect `MultipartRoutesTest` for route gating between allowed POST routes and file-upload routes.
- For doc-generation coverage, inspect `OpenApiSpecGeneratorTest` and `RestAPIDocGeneratorTest`.
