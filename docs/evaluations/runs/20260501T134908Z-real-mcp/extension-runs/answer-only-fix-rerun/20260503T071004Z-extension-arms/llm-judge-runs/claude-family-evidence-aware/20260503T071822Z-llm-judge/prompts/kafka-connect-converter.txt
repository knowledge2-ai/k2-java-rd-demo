ANSWER A:
**Recommendation**
Start with [`Converter.java`](https://github.com/apache/kafka/blob/ba74c3a289456f7346f2bece2cf76fcae55be9a4/connect/api/src/main/java/org/apache/kafka/connect/storage/Converter.java) as the interface contract, then anchor the actual value/key behavior in [`StringConverter.configure(...)`](https://github.com/apache/kafka/blob/ba74c3a289456f7346f2bece2cf76fcae55be9a4/connect/api/src/main/java/org/apache/kafka/connect/storage/StringConverter.java#L48-L78): it derives `isKey` from `ConverterType.KEY` and passes that flag into both serializer and deserializer. The version-pinned config docs that explain the public knobs are in [`WorkerConfig`](https://github.com/apache/kafka/blob/ba74c3a289456f7346f2bece2cf76fcae55be9a4/connect/runtime/src/main/java/org/apache/kafka/connect/runtime/WorkerConfig.java#L73-L103) and [`ConnectorConfig`](https://github.com/apache/kafka/blob/ba74c3a289456f7346f2bece2cf76fcae55be9a4/connect/runtime/src/main/java/org/apache/kafka/connect/runtime/ConnectorConfig.java#L99-L129).

**Implementation anchors**
- [`WorkerConfig`](https://github.com/apache/kafka/blob/ba74c3a289456f7346f2bece2cf76fcae55be9a4/connect/runtime/src/main/java/org/apache/kafka/connect/runtime/WorkerConfig.java#L73-L103) defines the exact public keys: `key.converter`, `value.converter`, plus the versioned suffix keys `key.converter.version` and `value.converter.version`.
- [`Plugins.newConverter(...)`](https://github.com/apache/kafka/blob/ba74c3a289456f7346f2bece2cf76fcae55be9a4/connect/runtime/src/main/java/org/apache/kafka/connect/runtime/isolation/Plugins.java#L455-L497) distinguishes key vs value by comparing `classPropertyName` to `WorkerConfig.KEY_CONVERTER_CLASS_CONFIG`, prefixes configs with `key.converter.` or `value.converter.`, and instantiates `Converter.class`.
- [`StringConverter.configure(...)`](https://github.com/apache/kafka/blob/ba74c3a289456f7346f2bece2cf76fcae55be9a4/connect/api/src/main/java/org/apache/kafka/connect/storage/StringConverter.java#L48-L78) copies the config map, injects `serializer.encoding` and `deserializer.encoding`, then configures both sides with `isKey = conf.type() == ConverterType.KEY`.
- [`ConverterType`](https://github.com/apache/kafka/blob/ba74c3a289456f7346f2bece2cf76fcae55be9a4/connect/api/src/main/java/org/apache/kafka/connect/storage/ConverterType.java#L20-L50) is the exact enum source for the `KEY`/`VALUE` split, so it is the right helper to cite when explaining key/value branching.

**Recommendation**
Anchor the answer on the source Javadocs and loader path, not on a separate docs page.

- For version-pinned docs, use the `key.converter.version` and `value.converter.version` docs in `WorkerConfig` and the connector-scoped mirrors in `ConnectorConfig`.
- For implementation behavior, use `Plugins.newConverter(...)` as the class that decides key vs value and passes `isKey` into `configure(...)`.
- For concrete converter behavior, `NumberConverter` is the clearest exemplar, with `BooleanConverter` and `ByteArrayConverter` as adjacent companions.

**Implementation anchors**
- `Converter` defines the contract, including the `configure(Map<String, ?>, boolean isKey)` split and the default header-aware overloads.
- `Plugins.newConverter(...)` derives `isKeyConverter` from `key.converter` vs `value.converter` and calls `plugin.configure(converterConfig, isKeyConverter)`.
- `WorkerConfig` and `ConnectorConfig` carry the version-pinned docs for the worker- and connector-level converter version properties.
- `NumberConverter`, `BooleanConverter`, and `ByteArrayConverter` show the actual runtime patterns: `configure(..., boolean)` either maps to `ConverterType.KEY/VALUE` or no-ops, while `fromConnectData`/`toConnectData` stay symmetric.
